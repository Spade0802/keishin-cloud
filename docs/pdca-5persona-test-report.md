## ペルソナ1: 行政書士 田中（経審専門・ベテラン 15年）

> 「公式結果と完全一致するかが最重要。1点でもズレたらクライアントに説明できない。」

### Plan
経審提出書PDFと決算書PDFをアップロードし、公式結果通知書のP点と完全一致するか検証。
特に技術職員数値(Z1)、X1テーブルの精度、W点の内訳を重点確認。

### Do（実操作 + コード検証）
1. ランディングページ → 「登録して始める」→ ログイン画面
2. ログイン後 → ダッシュボード → Trial画面(4ステップウィザード)
3. Step1: 第58期決算書.pdf アップロード → Gemini Vision AI自動OCR
4. Step2: 経審提出書.pdf アップロード → 業種データ自動抽出(電気/管/電気通信/消防)
5. Step3: W項目チェック、技術職員データ入力
6. Step4: 前期データ入力 → 計算実行

### Check（検証結果）

**✅ 正常動作（コード検証済み）:**
- X1テーブル: 1,375,760千円→X1=1067 ✅ (テストケースで検証済み)
- X21テーブル: 336,010千円→X21=810 ✅
- X22テーブル: 44,332千円→X22=687 ✅
- Z2テーブル: 688,475千円→Z2=1148範囲 ✅
- P計算式: `max(6, min(2160, round(0.25×X1 + 0.15×X2 + 0.2×Y + 0.25×Z + 0.15×W)))` ✅

**❌ 問題発見:**

| # | 深刻度 | 問題 | 詳細 |
|---|--------|------|------|
| 1 | 🔴致命的 | 技術職員数値が0のまま | 別紙二の20名リストから資格コード×講習フラグで計算が必要。`calculateTechStaffValueByIndustry()`を追加済みだがUI未統合 |
| 2 | 🟡重要 | X1の2年平均/当期選択が不透明 | 提出書では「2年平均と当期の大きい方」を選択するが、UIにどちらが使われているか表示がない |
| 3 | 🟡重要 | 元請完工高のUI入力が不足 | Z2計算に元請完工高が必要。完工高−下請で計算しているはずだが、元請を直接入力/確認する場所がない |

### Act
| 優先 | 改善 | 状態 |
|------|------|------|
| P0 | 別紙二→技術職員数値の自動計算 | ✅ エンジン実装済み、UI統合済み（worktree） |
| P1 | 2年平均 vs 当期の選択ロジック明示 | 未着手 |
| P1 | 元請完工高の表示/入力UI | 未着手 |

## ペルソナ4: IT担当 鈴木（SaaS評価担当・技術理解あり）

> 「信頼性・セキュリティ・パフォーマンスが評価基準。競合と比較して導入判断する。」

### Plan
技術スタック、API設計、エラーハンドリング、パフォーマンス、セキュリティを評価。

### Do（コード静的解析 + ブラウザ計測）
1. Next.js 16 + React 19 + Tailwind 4の構成確認
2. Gemini API呼び出しパターン確認
3. コンソールエラー/ネットワークエラー確認
4. 認証・DB設計確認
5. 初回ロード速度計測

### Check（検証結果）

**✅ 良い点:**
- Next.js 16(Turbopack): devサーバー起動1.5秒、初回ページロード10秒（コンパイル含む） ✅
- コンソールエラー: 0件（HMR connected のみ） ✅
- TypeScript: 全ファイルコンパイルエラー0 ✅
- テスト: 38テスト全パス ✅
- 認証: NextAuth 5 + bcrypt + JWT（堅牢） ✅
- Gemini API: temperature=0（OCR抽出時の再現性確保）✅
- 429フォールバック: Gemini Paid → Vertex AI 自動切替 ✅
- Consensus抽出: 2並列実行で合意値を採る（精度向上）✅

**❌ 問題発見:**

| # | 深刻度 | 問題 | 詳細 |
|---|--------|------|------|
| 1 | 🟡重要 | API呼び出し回数が多い | 1回の抽出で5 Gemini呼び出し + AI分析1回 = 6回。3-4テストで日次枠枯渇リスク |
| 2 | 🟡重要 | レスポンス時間が長い | PDF抽出15-30秒 + AI分析10-20秒 = 合計30-50秒。プログレスバーなし |
| 3 | 🟡改善 | Step途中のブラウザ閉じでデータ消失 | localStorageへの一時保存なし |
| 4 | 🟢確認済 | APIキーのサーバーサイド限定 | API RouteのServer Actionからのみ呼ばれており、クライアントに漏れない |
| 5 | 🟢確認済 | DB設計 | Drizzle ORM + PostgreSQL、simulations/scenarios/usersテーブルが適切に設計 |

### Act
| 優先 | 改善 | 状態 |
|------|------|------|
| P1 | 抽出処理のプログレスバー（5段階表示）| 未着手 |
| P1 | Step途中データのlocalStorage一時保存 | 未着手 |
